Page Table

運作原理

一個x86的page table是由2^20個page table entries組成，而每一個page table entry包含一個20位元的physical page number。 paging hardware需要利用virtual address的最高20位元來找到該虛擬位址在page table中，然後再把最高的20位元替換為對應的page table entry中的physical page number。 剩下的最低12位元會被paing hardware原樣的複製上去到physical address中。而這一個page table提供了2^12大小的內存。

虛擬跟實體的地址轉換有兩個步驟:

  1. 找page directory:
    page directory(4096-byte)，在每個page directory中有1024個分別指向一個其page table pages，而每一個page table含有1024個32位元的page table entry。 paging hardware就會使用虛擬位址的最高10位元來選擇page directory entry。
  2. 找page table:
    如果當下的page directory entry 存在的話，就會使用虛擬位址接下來的10個位元來從page table page中找出對應的page table entry。如果不存在的話，paging hardware就會拋出錯誤。

每一個page table entry都告訴了paging hardware其虛擬位址的使用權限。以下是其對於PTE所做的定義及宣告:

0833 #define PTE_P           0x001   // 表示PTE是否存在
0834 #define PTE_W           0x002   // 表示指令能否對page執行寫的動作，如果不能則執行讀跟取的動作
0835 #define PTE_U           0x004   // 此用來控制用戶process能否使用，不能的話只有內核可用
0836 #define PTE_PWT         0x008   // Write−Through 
0837 #define PTE_PCD         0x010   // Cache−Disable 
0838 #define PTE_A           0x020   // Accessed 
0839 #define PTE_D           0x040   // Dirty 
0840 #define PTE_PS          0x080   // Page Size 
0841 #define PTE_MBZ         0x180   // Bits must be zero

Virtual 跟 Physical Address 的安排


每一個process都有自己的page table，而user’s process的memory都由0開始排最多只能夠長到KERNBASE的地方，這樣只有2GB的空間。如果當process向XV6要求更多空間，XV6就只能提供使用從KERNBASE和0xFE00000之間的內存來使用，但是大部分的process是用不完整個內存空間的。

XV6在每個process都包含了kernel運行所需要的對應，而這些對應都出現在KERNBASE上。因此它將KERNBASE到KERNBASE+PHYSTOP的Virtual Address對應 0到PHYSTOP的Physical Address上。
如此處理的原因有以下兩點:

  1. kernel可以使用自己的指令和數據
  2. kernel有時候需要對Physical Address進行改寫

freerange是通過每一頁使用kfree實現將內存加入free list。分配器原本一開始是沒有內存的，但是經過kfree的調用使得可用的內存交給了分配器來管理。

使用者的address space

在exec中負責創建使用者部分系統的調用，它會根據文件系統中保存的某個文件來初始化用戶部分。

EXEC創建的第一個用戶程序為以下:

#objdump -p _init

_init:    file format elf32-i386

Program Header:
    LOAD off    0x00000054 vaddr 0x00000000 paddr 0x00000000 align 2**2
         filesz 0x000008c0 memsz 0x000008cc flags

program header filesz可能會比memsz來得小，在上面表示:filesz是2240bytes而memsz是2252bytes。所以allocuvm會分配足夠的內存來裝2252bytes的內容，但只從文件 /init 中讀取2240bytes的內容。
