Chapter 0   Operating system interfaces

       現在的電腦都是多工的，在多工的電腦上可以執行許多應用程式，可是當應用程式需要使用硬體時該如何運作呢？若多個應用程式需要使用硬體時又該如何決定先後順序與如何運作來讓電腦看起來像多工呢？這時候就由作業系統來處理這些問題，所有需要用到硬體的程式只要呼叫作業系統，作業系統就會幫你來完成這些事情，如此一來應用程式也就不用管要怎麼處理這些問題。
       那麼作業系統是如何運作的呢？通常透過使用者輸入指令後，執行getcmd的function，然後先fork出一個新的副本，fork的唯一差別在pid，這個是用來分辨該程式是fork出來的還是原本的，接著再用runcmd執行裡面的程式，通常執行程式還會用到exec的function：原先的程式會使用wait等待fork出來的程式完成再繼續執行接續的事。
       

↑(圖一) 一個cmd的大致流程

       有時候在執行時會需要讀或寫一些東西(通常是檔案)，可是這些東西的格式都不太相同該如何處理？這時就需要靠File descriptors幫我們處理這些東西的I/O；File descriptors記錄著該process需要用到的檔案名字與位置，並且對每個檔案提供三個接口，0為standard input(讀取用)，1為standard output(寫入用)，2為starndard error(輸出錯誤訊息)；例如cat function所做的功能就是將a檔案的東西丟到b檔案裡，此程式先利用read()並透過0的接口將檔案內的資料一個一個讀出，並且再利用write()並透過1的接口寫到另一個地方，當出現問題時就利用2的接口輸出錯誤訊息；如此就可以簡單地完成對檔案的操作，只需要透過那三個接口就可以完成與檔案相關的事情；要注意的是多個process所指到的某個檔案是共用同個offset，因為該檔案的offset是獨立於外部的(相對於process)，所以並不會因為是不同process的操作而有不同的offset。
       兩個process可能需要互相傳送資料，不想透過檔案，此時就可以使用pipe的機制來實行，將兩個process接上pipe的兩端，一個接0一個接1即可直接傳輸(File descriptors的0和1)，與透過檔案間接傳輸的差異在於處理完後需將該檔案刪除，並且使用pipe的方式不會受到檔案長度的限制，可以傳送任意長度的資料。
       一個作業系統裡面有許多檔案，分別放在不同位置，需要看到這些檔案的話雖然可以直接把那些檔案的路徑和檔名直接記起來然後直接找到，但檔案太多時就做不太到了，因此需要一個file system來幫我們管理(類似windows的檔案總管)；這個系統存在一個目錄，其實這個目錄是個特殊的文件，他是一個樹狀結構，往下連接到目錄或是檔案，目錄還可以再往下連接到其他目錄和檔案，而這個系統有幾個操作，分別是chdir、mkdir、mknod、open、close等指令，chdir可以切換現在的目錄位置，例如chdir(“/a”);就是將現在的目錄切換到根目錄下的a目錄，也可以使用chdir(“b”);意旨切換到現在目錄下的b目錄的位置，換言之就是進入到此目錄下的b目錄；mkdir則是新增一個目錄(類似windows的新增資料夾)；mknod則是新增一個檔案；open和close則是息息相關的，open可以開啟某個檔案並在後面加上不同的參數來決定這個檔案的模式(只能讀或只能寫等)，而close則是操作完檔案後將其關閉，跟系統說我已經做完了，可以給其他人用了；而每個檔案都有一個獨立的號碼稱為inode number，有時候我們會從別的地方link到該檔案(類似捷徑)，讓我們比較方便使用，而它的原始位址其實也是個link連結到硬體的位置，所以當有個檔案沒有任何link時，該檔案視同被刪除了，因為沒有人可以找到此檔案了，因此系統就知道要把這個檔案刪除了。

Chapter 1   Operating system organization

       在系統運作的時候，並不會把所有的東西都從記憶體裡搬出來，而是利用pointer的方式運作，一開始先建立一個virtual address space，所有pointer都會放在這裡，這是一個虛擬的空間，裡面所存放的東西都會指到記憶體中的其中一個位址，當需要用到資料的時候會先從virtual address space找到該資料的位址，然後再透過表格找到在實體記憶體的位址，然後再把資料取出。

↑(圖二) virtual address space與實體記憶體的關係

       當一些初始設定(上述的表格設置等)完成後，就會開始準備第一個要執行的process，他會先新增一個kernel stack，並且將process的相關資料存到這個stack，例如function return的位置等(如圖三)，當stack準備好後就開始將code複製到一個實體位置等待執行，並將virtual address space裡的pointer指到剛剛複製出來的位置，接者這個process就已經準備好可以run了，此時會將p->state設定為RUNNABLE。
↑(圖三) kernel stack

       接著scheduler就會開始找有哪個process的p->state是RUNNABLE的，當找到後會開始執行，並將該process的p->state設定為RUNNING；然後將相關的register的值存到stack避免被修改，等process結束後再放回register裡，接著把process搬到CPU後開始執行。
       然後第一個process裡會執行一個system call叫做exec，執行另一個程式，並且把另一個程式的搬到CPU裡執行，同樣的也會經過前幾段所寫的事情，當該程式結束後又會跳回來第一個process，然後再找下一個exec的程式，重複進行這些動作，就使一個作業系統運作起來了。

Chapter 4   Locking

       在多處理器的機器下，每個處理器都會執行一段程式，但如果有兩個程式需要用到同一個東西或變數的話，那麼該怎麼做才不會讓資料有不確定性，或者說該怎麼樣才可以確保此資料一次只會被一個程式修改，這時候我們就使用到一個方法 – Locking，當要有程式要進行修改時先對該資料進行lock，讓所有人都不能對此資料進行操作，直到該程式unlock時其他程式才可以對她進行操作；利用這個方式就可以確保該資料一次只會被一個程式做修改。
       接著以下這段code是一個基本的link list的新增，將一個新的node新增在原本的list的前面，這裡的node的結構很簡單(此處以struct list展示)，只有一個data，next則是指向下一個位置的pointer 




















struct node{
    int data;
    struct node *next;
};

struct node *list = 0;

void
insert(int data)
{
    struct node *l;

    l = malloc(sizeof *l);
    l->data = data;
    l->next = list;
    list = l;
}
       以上的範例在邏輯上是正確的，但當有兩個以上的CPU同時執行這段code的話就有可能發生data racing的問題；當兩個CPU一起執行時這個程式時，兩邊會先各自製造一個node，各自塞好資料後指到同一個list的開頭，接著在第16行的時候就會發生問題了，當要將list = l;的時候，後面執行的會把前面的蓋掉；正常來說我們希望執行兩次就是新增兩個node到list裡，可是因為rasing的關係，所以不會變成我們想要的情況。
       為了解決這問題，我們將上面的範例改成如下所示，新增了幾行code(沒有行號的那幾行)，在開始時先使用acquire向系統要求使用這個區塊並且將listlock這個變數鎖起來(標記)，當有其他CPU要執行相同的code時會先去看該變數是否可以被使用，如果沒有才會繼續執行，在完成後使用release將listlock還原，讓其他CPU可以使用，如此一來就可以避免rasing的問題了。 
























struct node{
    int data;
    struct node *next;
};

struct node *list = 0;
struct lock listlock;

void
insert(int data)
{
    struct node *l;

	acquire(&listlock);

    l = malloc(sizeof *l);
    l->data = data;
    l->next = list;
    list = l;
	
	release(&listlock);
}
       接著來講講acquire如何運作的，以下是acquire的code，主要的功能在第10行，他會嘗試取的lk這個的控制權，利用lk->locked來檢查是否被lock了，如果是則值為1；但只有這樣還不夠，如果只是用單純的if來判斷的話，他自己就會發生rasing的問題了，因此我們需要xchg來幫助我們解決這個問題，xchg他在做的事情就是將兩個參數交換，例如第十行是把lk->locked的值寫成1，如果這個成功改變的話xchg會回傳1否則會回傳0，所以當成功改變(亦指0變成1)時，將會繼續執行下面的事，否則會停在while繼續等待，改變成功時就會將lk->cpu的值改成搶到控制權的cpu的值。 






















void
acquire(struct spinlock *lk)
{
	pushcli(); // disable interrupts to avoid deadlock.
	
	if(holding(lk))
		panic("acquire");
	
	// The xchg is atomic.
	while(xchg(&lk−>locked, 1) != 0);
	
	// Tell the C compiler and the processor to not move loads or stores
	// past this point, to ensure that the critical section’s memory
	// references happen after the lock is acquired.
	__sync_synchronize();
	
	// Record info about lock acquisition for debugging.
	lk−>cpu = mycpu();
	getcallerpcs(&lk, lk−>pcs);
}
       雖然有lock的保護，但不是有他一切都OK，當同時有個鎖被多個CPU搶用時要小心這個鎖的對應的先後關係，例如有兩個CPU都要使用A、B兩個lock，可是如果這兩個CPU的要求順序一個是A->B、另一個是B->A的話，會造成互相等待的情況發生，這種情況我們稱為deadlock，雙方都在等待對方結束而造成系統停擺，因此我們在呼叫lock的時候前後順序必須是相同的。
       有時候程式會因為某些原因中斷(例如sleep)，此時若該程式持有某些lock的控制權的話，當其他程式嘗試使用這個lock時就會一直等待，可是該程式已經中斷了，不知何時會還回來(因為不會完成)，因此造成deadlock，在xv6裡，xv6在程式要中斷時必須將所有lock還回去才可以進行中斷，而判斷還有幾個所要還的方式是利用popcli和pushcli這兩個function，在上面的acquire的function裡的第四行有看到pushcli這個函式，他會記錄一個該CPU有幾個lock並且增加mycpu()->ncli的數量；當呼叫cli()這個function(中斷的function)時會檢查mycpu()->ncli的數值是否為0，若不是的話則不會允許這個程式中斷，必須將所有的lock還完後才可以使用cli()，對應著pushcli()的是popcli()，存在於release()裡
       系統為了效能有時候會將沒有影響的程式碼重新排序，讓整個程式可以更快速地完成，例如上面範例裡的insert function中的14和15行其實換位置不會影響程式，但有時候不希望compiler幫你換位置(避免有一些不必要的錯誤)，因此我們使用__sync_synchronize()來跟compiler說這個不要幫我亂換位置，確保不會有問題；這個function用在acquire和release裡，因為這兩個是lock的很關鍵的東西，如果有compiler亂換位置的不確定性，造成一些不必要的問題。
       而有時候需要lock很長的時間，像是file system在進行讀寫的時候，所以xv6提供了一個新的lock方式給file system，稱作sleeplock，其實做是使用acquiresleep來實作，其差別在於acquiresleep是lock要lock的上一層，而不是直接在等要lock的那一個，像是一個排隊的觀念，當現在無法取的要的那個lock時，就先用sleep()等一下，過一段時間後再問一次，反覆直到可以拿到那個lock為止；用這種方式的最大差別在於當在等的時候其他process是有可能使用到這個鎖的(在sleep時要求使用這個lock而且前一個也剛好用完了)。
